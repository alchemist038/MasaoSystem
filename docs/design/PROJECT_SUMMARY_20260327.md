# プロジェクト要約

作成日: 2026-03-27

## 1. このプロジェクトの目的

このプロジェクトの中心は、うさぎのまさおの長時間ライブ録画から、見どころを見つけてショート動画を作り、YouTube に投稿することです。

作業全体としては、次の流れを目指しています。

1. 長時間ライブ動画を録画する
2. 処理しやすい形に軽量化する
3. 動きのある区間を見つける
4. まさおが映っている区間に絞る
5. ショート動画を作る
6. YouTube に投稿または予約する
7. 投稿後の動画をまとめて整備する
8. 将来的にはライブチャット / スーパーチャット返信系も別ラインとして持つ

## 2. 機能ごとのライン

このプロジェクトは、1 本のスクリプト群というより、次の複数ラインの集合として見るのが自然です。

### 2.1 録画前処理ライン

目的:

- 長時間録画を後工程で処理しやすい素材に変える

例:

- 360 化
- `raw.mkv`
- `proxy_360.mp4`

### 2.2 ΔY 動体検出ライン

目的:

- 録画全体から、動きのある区間を粗く拾う

例:

- `showinfo`
- `meanY`
- `deltaY`
- イベント候補抽出

### 2.3 YOLO フィルタライン

目的:

- 候補区間の中から、まさおが実際に映っている区間に絞る

例:

- `raw_yolo.jsonl`
- 20 秒候補生成
- スコア付け
- ピック

### 2.4 PTZ / モーション由来ショート生成ライン

目的:

- PTZ で中央に入りやすくなった映像を使い、より良い切り抜き位置でショートを作る

例:

- 20 秒固定ショート
- `crop_x`
- metadata 生成
- render
- upload

### 2.5 投稿後メンテナンスライン

目的:

- すでにアップロード済みの動画を後からまとめて整える

例:

- タイトル更新
- 説明欄更新
- 公開予約
- 再生リスト追加

### 2.6 チャット / スーパーチャット返信ライン

目的:

- 特定ワード返信
- スーパーチャット返信
- 配信中補助

これはショート生成ラインとは別のプロジェクトとして扱うのがよいです。

## 4. 整理で最優先に守ること

整理するときの最重要ルールは、いま使っている入口を壊さないことです。

具体的には次の 2 つです。

1. `D:\OBS\REC\scripts\youtube\yolo\WIN\UPLOAD_RUNBOOK.md`
2. `D:\OBS\REC\work\UPDATE_SHORTS_RUNBOOK.md`

このため、整理の基本ルールは次の通りです。

- いきなり current を移動しない
- いきなり legacy を消さない
- 先に `current / next / legacy` を定義する
- 次に絶対パスを設定化する
- 最後に物理移動を考える

## 5. おすすめの方向性

一番安全な整理方向は次です。

1. `scripts\youtube\yolo\WIN` を current として固定する
2. `work` を post-publish 専用ラインとして固定する
3. `scripts\WIN_YOLO_PTZ_20S` を next として育てる
4. Linux 旧ラインと `WIN` を legacy として扱う
5. チャットボット系はショート生成と分ける

## 6. いまの結論

このプロジェクトは 1 本のコードツリーではありません。

実態としては次の集合です。

- 現在使っている Windows ショート生成ライン
- 現在使っている投稿後整備ライン
- 古い Linux 共有キュー型ライン
- 古い Windows ブリッジ
- 次世代候補の Windows ライン
- 学習・研究・モデル資産
- まだ独立整理されていないチャットボット構想

したがって、最初にやるべきことは大移動ではなく、役割を明確にして守る線を決めることです。
