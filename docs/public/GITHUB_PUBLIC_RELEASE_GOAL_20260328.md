# GitHub Public Release Goal

作成日: 2026-03-28

## 1. 最上位目標

このプロジェクトの最終目標は、
GitHub 公開を前提とした整理を行い、
低スペック環境でも動く自動処理パイプラインの構成と思想を公開することです。

以後の作業では、この目標を最高優先度で扱います。

## 2. 公開で重視するもの

公開時に主役にするのは、コード量ではなく次の要素です。

- README
- 全体設計
- 実運用で使われている構造
- state 管理
- queue 設計
- immutable な考え方

つまり、
「何をどう作ったか」よりも
「どういう構造で安定運用しているか」を伝えることを優先します。

## 3. 公開しないもの

GitHub 公開版には、次を含めません。

- API key
- token
- 実運用の秘密情報
- 実運用 prompt
- 完全な本番ロジック
- 個別運用に強く依存した private 情報

## 4. 公開版でやること

1. README を主役として構成する
2. フォルダ構造を整理する
3. 公開用に不要なファイルを除外する
4. サンプルデータを用意する
5. 動く最小構成にする

## 5. 制約

- 本番環境は壊さない
- コピーして別ディレクトリで作業する
- シンプルで理解しやすい構造にする

## 6. 完成条件

- README だけで全体像が理解できる
- 初見の人が構造を理解できる
- GitHub にそのまま公開できる状態

## 7. 既存の再構築方針との関係

既存の `D:\MasaoSystem` 構想や current / next / legacy の整理は、
この GitHub 公開目標のための前段作業として扱います。

つまり、

- ローカル運用を壊さず
- 実運用の構造を抽出し
- 公開版として説明可能な最小構成へ落とす

ための整理であると解釈します。

## 8. 今後の判断基準

今後の作業で迷った場合は、次の順に優先します。

1. GitHub 公開時に説明しやすいか
2. README 主体で理解できるか
3. 実運用の構造を壊さず抽出できているか
4. 本番環境を壊していないか
5. 低スペック環境向けの思想が見えるか

## 9. 次の具体作業

次に着手すべき作業は次の通りです。

1. 公開対象として何を残すかを決める
2. GitHub 公開版の新しい作業先を別ディレクトリに作る
3. README の目次を先に作る
4. 公開版に必要な最小フォルダ構成を定義する
5. sample data と mock config の方針を決める
6. 除外対象を明文化する

## 10. 一言で言うと

最終的に目指すのは、
「README を読めば、低スペック環境でも動く自動処理パイプラインの構造思想が理解できる GitHub 公開用ミニマム構成」
です。
